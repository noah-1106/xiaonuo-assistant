# 微信支付对接实现计划

## 一、当前系统分析

### 1. 现有实现

* **前端**：PlanPage组件展示套餐列表，有"立即购买"按钮，但无购买流程
* **后端**：
  * Order模型：支持订单创建、查询和状态更新
  * Plan模型：支持套餐管理，包含价格、时长等信息
  * User模型：包含subscription字段，管理用户套餐
* **API**：订单和套餐相关API已实现，但缺少支付功能

### 2. 限制条件

* 域名备案未完成，无回调地址
* 需采用轮询方式处理支付结果
* 网站使用，建议采用Native支付（生成二维码，用户扫码支付）

* **创建订单**：修改createOrder API，支持传入购买数量
* **生成支付二维码**：添加generateWechatNativeQR API，调用微信Native支付API生成二维码

#### 2.3 配置管理

* 添加微信支付配置文件，存储appid、mchid、apiKey等参数

### 3. 支付流程设计

1. **用户选择套餐** → 2. **点击"立即购买"** → 3. **弹出订单弹窗，选择购买数量** → 4. **计算总金额，提交订单** → 5. **弹出支付方式选择** → 6. **选择微信支付** → 7. **后端调用微信Native支付API生成二维码** → 8. **前端展示二维码，同时轮询检查支付结果** → 9. **用户扫码支付** → 10. **轮询检测到支付成功** → 11. **后端更新订单状态和用户套餐** → 12. **前端显示支付成功提示，更新用户套餐信息**

## 三、实现细节

### 1. 前端组件设计

#### OrderModal.tsx

* 表单字段：套餐信息、购买数量（默认1）、总金额（自动计算）
* 功能：确认订单，提交订单请求

#### PaymentMethodModal.tsx

* 支付方式：微信支付（Native支付）
* 功能：选择支付方式，请求支付二维码

#### PaymentQRModal.tsx

* 显示微信Native支付二维码
* 实现轮询检查支付结果（每3秒一次）
* 支付成功/失败提示

### 2. 后端API设计

#### POST /orders

* **功能**：创建订单
* **参数**：planId, quantity（购买数量，默认1）
* **返回**：订单信息

#### POST /orders/:id/pay

* **功能**：生成微信Native支付二维码
* **参数**：paymentMethod（wechat）
* **返回**：支付二维码链接，订单ID，二维码过期时间

#### GET /orders/:id/pay-status

* **功能**：查询支付状态
* **返回**：支付状态（pending/paid/cancelled）

#### PUT /orders/:id/status

* **功能**：更新订单状态
* **参数**：status, paymentTime
* **逻辑**：
  * 若状态为paid，计算套餐有效期
  * 若用户已有套餐且未过期，叠加时长
  * 更新用户subscription信息

### 3. 套餐叠加逻辑

```javascript
// 伪代码

  newEndDate.setMonth(newEndDate.getMonth() + (plan.duration * quantity));
```

## 九、注意事项

1. 由于域名备案未完成，使用轮询方式处理支付结果，生产环境建议使用回调地址
2. 微信支付配置信息需妥善保管，避免泄露
3. 轮询间隔设置为3秒，避免频繁请求
4. 支付超时时间设置为300秒，超时后提示用户重新支付
5. 套餐叠加仅支持同一类型套餐，不同类型套餐需重新计算
